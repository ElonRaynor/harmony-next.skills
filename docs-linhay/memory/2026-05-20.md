## Issue 10: DevEco Emulator 非交互策略

- 处理 GitHub issue #10：`Make DevEco Emulator automation policy non-interactive for long-running agents`。
- 决策：harmony-next skill 不做授权或人工确认拦截；用户默认拥有完整执行权限。
- 决策：`policy` 是执行模式、产物目录和脱敏契约，不是授权门槛。
- 决策：machine-readable `blocked` 只表示缺少 target、artifactDir、redactionPolicy、timeout、命令参数等客观运行配置。
- 决策：`break-glass` 是刷写、格式化、清数据、root/daemon 等系统级动作的风险标签，不是授权 gate。
- 验证：`python3 -m unittest discover -s harmony-next/tests -p 'test_*.py' -v` 通过，14 个测试成功。

## Issue 11: Empty Ability 最小工程模板

- 处理 GitHub issue #11：新增 `harmony-next/references/templates/empty-ability-app/`，作为可复制 HarmonyOS NEXT Empty Ability smoke fixture。
- 决策：模板基于本机 DevEco Studio `New Project` 与 `Empty Ability` 官方模板瘦身，保留 Stage 模型、`EntryAbility`、`pages/Index` 和基础资源。
- 决策：默认 `bundleName=com.example.emptyability`、module `entry`、ability `EntryAbility`、SDK `5.0.0(12)`，适配已有 Neptune/Triton Harmony 测试上下文。
- 决策：页面暴露 `Harmony Smoke Ready`、`smoke-title`、`smoke-counter`、`smoke-increment`，方便 HDC、`uitest dumpLayout`、截图和日志 smoke 自动化定位。
- 决策：模板不携带本机签名材料、绝对路径、IDE 缓存或 `${...}` 向导占位符。
- 验证：reference check/audit、全量 unittest 15 项、targeted metadata unittest 8 项、`git diff --check` 通过；模板副本 `ohpm install` 通过。
- 风险：本机 DevEco SDK 仅有 HarmonyOS 6.0.2 / API 22，缺少模板默认 API 12 组件，Hvigor assemble 在副本中报 `SDK component missing`，需要安装 API 12 SDK 后做完整 HAP 构建验证。
- 复测：使用 HarmonyOS 6.0.2 / API 22 时，`DEVECO_SDK_HOME` 应指向 SDK 根目录而不是 `sdk/default`；API 22 schema 要求 `AppScope/app.json5` 声明 app `icon`，Ability 声明 `icon` 和 `startWindowIcon`，已补充官方 `app_icon.png` 与 `$media:app_icon`。
- 结果：临时副本改为 `6.0.2(22)` 后，`DEVECO_SDK_HOME=/Applications/DevEco-Studio.app/Contents/sdk hvigorw --mode module -p module=entry@default assembleHap` 通过，生成 `entry-default-unsigned.hap`；无签名配置警告符合模板不携带签名材料的边界。
